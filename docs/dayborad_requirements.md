# dayborad 要件定義書

本プロダクトの正式表記は `dayborad` とする。名称の揺れを避けるため、本書ではこの表記に統一する。

## 1. 概要

**dayborad** は、PC画面を「その日の仕事ノート1枚」として使う、日次業務管理アプリである。

一般的なタスク管理アプリのようにプロジェクトや長期目標を細かく管理するのではなく、**今日やること、仕事の詰まり、1日の振り返り、会議・打ち合わせ・会話メモ**を、日付単位で記録・整理することを目的とする。

アプリ起動時は、画面いっぱいに以下の3領域を表示する。

1. TODO
2. 障害・詰まり
3. 振り返り

ユーザーが会議メモや会話メモを書きたい場合は、ショートカットキーでノートモードに切り替える。ノートモードでは、以下の領域が画面いっぱいに表示される。

4. 会議・打ち合わせ・会話メモ

つまり、dayborad は **「仕事整理モード」と「ノートモード」を切り替える、1日1枚の仕事ノートアプリ**である。

---

## 2. コンセプト

### 2.1 基本思想

PC画面そのものを、1枚の仕事ノートとして扱う。

- 通常時は、今日の仕事を整理する。
- 会議や会話の記録が必要なときだけ、ノートを開く。
- ノートに書いた内容のうち、必要なものだけを TODO や障害に変換する。
- 1日の終わりに、TODO と障害を見ながら振り返る。

### 2.2 メタファー

実際の紙ノートのように、以下の感覚で使えることを重視する。

| モード | ノート上のイメージ | 内容 |
|---|---|---|
| 仕事整理モード | ノートの表面 | TODO、障害・詰まり、振り返り |
| ノートモード | ノートの裏面 / メモページ | 会議・打ち合わせ・会話メモ |

### 2.3 アプリの役割

このアプリは、プロジェクト管理ツールではない。

目的は、以下を日次で整理することである。

- 今日やること
- 今日の仕事を止めているもの
- 会議や会話で発生したメモ
- 今日できたこと
- 今日止まったこと
- 明日の一手

---

## 3. 対象ユーザー

### 3.1 主なユーザー

- 日々の仕事で複数の TODO を抱えるビジネスパーソン
- 会議や打ち合わせが多い人
- 口頭やチャットで発生した依頼を忘れやすい人
- 既存のタスク管理ツールが重いと感じている人
- 紙ノート的な運用をデジタル化したい人

### 3.2 想定利用環境

- PC利用を主軸とする
- キーボード操作中心
- MVPはPC向けの利用体験を優先して提供する
- モバイル対応は初期スコープ外
- 将来的なWeb、デスクトップ、モバイル対応を見据え、UI、ドメインロジック、API、データモデルを分離しやすい構成にする

---

## 4. 解決したい課題

### 4.1 現状の課題

多くのタスク管理ツールでは、以下の問題が起きやすい。

- TODO は登録するが、なぜ進まないのかが残らない
- 会議メモと TODO が分断される
- 口頭で発生した依頼が流れてしまう
- 日々の振り返りが習慣化しない
- プロジェクト管理ツールが多機能すぎて毎日開かなくなる
- その日に必要な情報だけを見たいのに、長期タスクや過去情報が混ざる

### 4.2 dayborad が解決すること

- その日の TODO を1画面で把握できる
- TODO が止まっている理由を明確にできる
- 会議・会話メモをすぐに開いて書ける
- メモから TODO や障害に変換できる
- 1日の終わりに振り返りを残せる
- 日付ごとに仕事のログが残る

### 4.3 MVPの目標と成功指標

MVPでは、機能を作り切ることだけでなく、ユーザーが日次の仕事整理に継続して使えるかを検証する。

#### 目標

1. ユーザーが朝から夕方まで、1日1枚の仕事ノートとして使える。
2. 会議・会話メモから TODO や障害へ、作業を止めずに整理できる。
3. 入力内容が失われず、保存操作を意識せずに使える。

#### 成功指標

新規プロダクトのため、MVPリリース前のプロダクト利用ベースラインは 0 とする。ユーザーテストまたは限定配布で計測する。

| 指標 | ベースライン | MVP目標 | 判定タイミング |
|---|---:|---:|---|
| 初回利用後7日以内に3日以上 `DayNote` を作成・編集したユーザー率 | 0% | 40%以上 | 限定配布後2週間 |
| 1日の中で TODO、障害・詰まり、振り返りのうち2領域以上を入力した日次ノート率 | 0% | 60%以上 | 限定配布後2週間 |
| ノートモードの行を TODO または障害へ変換したユーザー率 | 0% | 50%以上 | 限定配布後2週間 |
| 自動保存失敗により入力を失った報告件数 | 未計測 | 0件 | MVP期間中 |
| モード切り替え操作が遅い・分かりにくいというユーザー報告率 | 未計測 | 20%未満 | ユーザーテスト後 |

#### 非目標

- 個人の長期タスクやプロジェクト全体を管理すること。
- チームで共有する業務管理基盤になること。
- AI要約や外部連携で自動整理すること。

## 5. スコープ

### 5.1 MVPで実装する範囲

MVPでは、以下を実装対象とする。

- 日付ごとの仕事ノート作成
- 仕事整理モード
  - TODO
  - 障害・詰まり
  - 振り返り
- ノートモード
  - 会議・打ち合わせ・会話メモ
- 仕事整理モードとノートモードのショートカット切り替え
- ノートモードの選択行を TODO 化
- ノートモードの選択行を障害化
- キーバインド設定
  - 標準キーバインド
  - VimキーバインドのMVP範囲
- 自動保存
- 前日 / 翌日 / 今日への移動
- 未完了 TODO の翌日持ち越し

### 5.2 MVPでは実装しない範囲

以下は初期スコープ外とする。

- 長期目標管理
- 中期目標管理
- プロジェクト管理
- ガントチャート
- チーム共有
- 権限管理
- カレンダー連携
- AI要約
- スマホ最適化
- 通知機能
- 複雑なタグ管理
- 外部チャット連携
- コマンドパレット
- ノート行の振り返りへの直接送信
- 時刻つきメモ見出しのショートカット挿入
- Vimキーバインドの高度な編集操作
  - `gg` / `G`
  - `A` / `o` / `O`
  - `dd`
  - `u` / `Ctrl + r`
  - `/` / `n` / `N`
- ユーザーによる個別キーマップ編集

## 6. 画面構成

## 6.1 全体構造

1つの日付に対して、1つの `DayNote` を持つ。

`DayNote` には、以下の2つの表示モードがある。

| モード | 表示内容 |
|---|---|
| work | ① TODO / ② 障害・詰まり / ③ 振り返り |
| note | ④ 会議・打ち合わせ・会話メモ |

## 7. 機能要件

## 7.1 日付ごとの仕事ノート

### 概要

日付単位で1つの仕事ノートを持つ。

### 要件

- アプリ起動時は当日のノートを表示する。
- 当日のノートが存在しない場合、自動生成する。
- 前日 / 翌日に移動できる。
- 過去の日付のノートも閲覧・編集できる。
- 日付ごとに TODO、障害、振り返り、ノート本文を保持する。

---

## 7.2 今日のテーマ

### 概要

その日の仕事の主目的を短く入力できる。

### 要件

- 各 `DayNote` に1つのテーマを設定できる。
- 未入力でも利用可能とする。
- ヘッダーに表示する。
- 仕事整理モード、ノートモードの両方で表示する。

### 入力例

```text
A社提案を前に進める
午前中に見積を終わらせる
会議後の宿題を整理する
```

---

## 7.3 ① TODO

### 概要

今日やることを管理する領域。

### 要件

- TODOを追加できる。
- TODOを完了できる。
- TODOを未完了に戻せる。
- TODOの表示順を変更できる。
- TODOは日付に紐づく。
- ノートモードからTODO化された内容もここに追加される。
- 未完了TODOは翌日に持ち越せる。

### 表示例

```text
① TODO
□ 見積作成
□ 田中さん確認
□ 議事録送付
```

### ステータス

| ステータス | 意味 |
|---|---|
| todo | 未完了 |
| done | 完了 |
| carried | 翌日に持ち越し済み |

---

## 7.4 ② 障害・詰まり

### 概要

仕事が進まない理由、TODOを止めている要因を記録する領域。

### 要件

- 障害・詰まりを追加できる。
- 障害・詰まりを編集できる。
- 障害・詰まりを解消済みにできる。
- TODOに紐づけられる。
- TODOに紐づかない障害も登録できる。
- ノートモードから障害化された内容もここに追加される。

### 表示例

```text
② 障害・詰まり
・A社回答待ち
・仕様不明点あり
・部長承認待ち
```

### 障害カテゴリ案

初期実装では任意入力でよい。将来的には以下の分類を付与できる。

| カテゴリ | 内容 |
|---|---|
| 情報不足 | 仕様、金額、納期などが不明 |
| 相手待ち | 顧客、上司、同僚の返答待ち |
| 判断待ち | 承認、方針決定、優先順位待ち |
| 作業分解不足 | 何から始めるか不明 |
| 時間不足 | 着手時間を確保できていない |
| 自分起因 | 後回し、確認漏れ、理解不足 |

---

## 7.5 ③ 振り返り

### 概要

1日の終わりに、成果・詰まり・明日の一手を記録する領域。

### 要件

- 以下の3セクションを持つ。
  - できたこと
  - 止まったこと
  - 明日の一手
- 各セクションは自由入力とする。
- 未入力でも利用可能とする。
- 夕方以降にアプリを開いた場合、振り返り欄を軽く強調してもよい。

### 表示例

```text
③ 振り返り

できたこと
・見積は完了

止まったこと
・確認依頼が遅い

明日の一手
・明日は朝に確認
```

---

## 7.6 ④ 会議・打ち合わせ・会話メモ

### 概要

会議、打ち合わせ、雑談、口頭依頼、自分用メモを自由に記録する領域。

### 要件

- ショートカットでノートモードを開く。
- ノートモードでは本文エリアを画面いっぱいに表示する。
- 自由入力できる。
- 時刻つきの見出しを手入力できる。
- 選択行をTODO化できる。
- 選択行を障害化できる。
- TODO化、障害化した行にはマークを付ける。
- ノート本文は日付に紐づく。

### 表示例

```text
④ 会議・打ち合わせ・会話メモ

10:00 A社定例
- 決定事項：来週までに見積提出
- 宿題：単価表を確認
- TODO化：見積作成

13:30 田中さんと会話
- 仕様不明点あり
- 部長承認待ち
```

---

## 7.7 モード切り替え

### 概要

仕事整理モードとノートモードをショートカットで切り替える。

### 要件

- 初期表示は仕事整理モードとする。
- `Ctrl / ⌘ + J` でノートモードに切り替える。
- ノートモード中に `Ctrl / ⌘ + J` を押すと仕事整理モードに戻る。
- ノートモード中に `Esc` を押すと仕事整理モードに戻る。
- モード切り替え時も入力内容は自動保存される。
- ノートモードから戻った際、追加されたTODOや障害を短時間ハイライトする。

### 状態定義

```ts
type ViewMode = 'work' | 'note'
```

---

## 7.8 ノートからTODO化

### 概要

ノートモードで選択した行を、TODOとして①に追加する。

### 要件

- ノート本文内の行を選択できる。
- `Ctrl / ⌘ + Enter` で選択行をTODO化する。
- TODO化された内容は①TODOに追加される。
- 発生元として、元のノート行への参照を保持する。
- TODO化済みの行にはマークを付ける。
- 同じ行を再度TODO化する場合は重複確認を行う。

### MVPでの行変換ルール

- TODO化時点の行本文をスナップショットとして保持する。
- TODO化後にノート本文を編集しても、作成済みTODOの本文は自動変更しない。
- 変換済みマークは、変換時点の `noteEntryId`、正規化済み行本文、行ハッシュをもとに表示する。
- 同じ `noteEntryId` 内で同一内容の行を再度TODO化しようとした場合は、既存TODO候補を表示して重複確認を行う。
- 重複確認でキャンセルした場合、新しいTODOは作成しない。
- 重複確認で作成を選んだ場合、別TODOとして作成し、発生元スナップショットも別に保持する。
- ノート本文の編集により元行を特定できなくなった場合でも、TODO側には変換時点の行本文を表示できるようにする。

### 例

ノート本文：

```text
- 宿題：単価表を確認
```

TODO化後：

```text
□ 単価表を確認
```

---

## 7.9 ノートから障害化

### 概要

ノートモードで選択した行を、障害・詰まりとして②に追加する。

### 要件

- ノート本文内の行を選択できる。
- `Ctrl / ⌘ + Shift + B` で選択行を障害化する。
- 障害化された内容は②障害・詰まりに追加される。
- 発生元として、元のノート行への参照を保持する。
- 障害化済みの行にはマークを付ける。
- 障害化でも、TODO化と同じ行変換ルールを適用する。

### 例

ノート本文：

```text
- 部長承認待ち
```

障害化後：

```text
・部長承認待ち
```

---

## 7.10 未完了TODOの翌日持ち越し

### 概要

当日中に完了しなかったTODOを、翌日のノートに引き継ぐ。

### 要件

- 未完了TODOを翌日に持ち越せる。
- 持ち越し元の日付を保持する。
- 持ち越し後、元TODOには持ち越し済みマークを付ける。
- 同じTODOを重複して持ち越さない。
- 翌日のノートが未作成の場合、自動生成する。

### 表示例

当日：

```text
□ 田中さん確認　→ 7/9へ持ち越し済み
```

翌日：

```text
□ 田中さん確認　7/8から持ち越し
```

---

## 7.11 自動保存

### 概要

ユーザーが保存操作を意識しなくてよいよう、入力内容は自動保存する。

### 要件

- TODO、障害、振り返り、ノート本文、今日のテーマは自動保存する。
- 入力後、短い遅延を置いて保存する。
- 保存状態を画面右上などに小さく表示する。
- 保存失敗時はエラーを表示する。

### 保存状態表示例

```text
保存済み
保存中...
保存できませんでした
```

## 8. ショートカットキー

## 8.1 基本ショートカット

| 操作 | Windows | Mac | 内容 | MVP範囲 |
|---|---:|---:|---|---|
| ノートモード切替 | Ctrl + J | ⌘ + J | ①②③ ⇄ ④ を切り替える | MVP |
| ノートモードから戻る | Esc | Esc | ④から①②③へ戻る | MVP |
| キーバインドガイド | ? | ? | 現在のモードで使えるショートカット一覧を表示する | MVP |
| コマンドパレット | Ctrl + K | ⌘ + K | 操作検索 | Post-MVP |
| 今日へ戻る | Ctrl + T | ⌘ + T | 今日の日付に戻る | MVP |
| 前日へ | Alt + ← | Option + ← | 前日のノートへ | MVP |
| 翌日へ | Alt + → | Option + → | 翌日のノートへ | MVP |

> **キーバインドガイド（`?`）の補足:** 仕事整理モード・ノートモードのどちらでも、標準キーバインド・Vimキーバインドのどちらでも有効。テキスト入力欄（input/textarea/CodeMirror）にフォーカス中は `?` を文字入力として扱い、ガイドは開かない（入力欄以外にフォーカスがある状態でのみ開く）。ヘッダーのヘルプアイコンからも開ける。`Esc` / 背景クリック / もう一度 `?` で閉じる。

## 8.2 仕事整理モード内ショートカット

| 操作 | Windows | Mac | 内容 | MVP範囲 |
|---|---:|---:|---|---|
| TODOへ移動 | Ctrl + 1 | ⌘ + 1 | ①TODOにフォーカス | MVP |
| 障害へ移動 | Ctrl + 2 | ⌘ + 2 | ②障害・詰まりにフォーカス | MVP |
| 振り返りへ移動 | Ctrl + 3 | ⌘ + 3 | ③振り返りにフォーカス | MVP |
| TODO追加 | Ctrl + Enter | ⌘ + Enter | TODOを追加 | MVP |

## 8.3 ノートモード内ショートカット

| 操作 | Windows | Mac | 内容 | MVP範囲 |
|---|---:|---:|---|---|
| 選択行をTODO化 | Ctrl + Enter | ⌘ + Enter | ①TODOへ送る | MVP |
| 選択行を障害化 | Ctrl + Shift + B | ⌘ + Shift + B | ②障害・詰まりへ送る | MVP |
| 選択行を振り返りへ送る | Ctrl + Shift + R | ⌘ + Shift + R | ③振り返りへ送る | Post-MVP |
| 時刻つきメモ追加 | Ctrl + Shift + M | ⌘ + Shift + M | 現在時刻の見出しを追加 | Post-MVP |

## 8.4 キーバインドモード

ユーザーは、操作体系として以下の2種類を切り替えられる。

| モード | 内部値 | 内容 |
|---|---|---|
| 標準キーバインド | `standard` | 一般的なPCアプリに近いショートカット体系 |
| Vimキーバインド | `vim` | Vim風の移動・編集・コマンド操作を利用する体系 |

本要件定義書では、アプリの表示状態である `work / note` を「表示モード」、キーボード操作体系である `standard / vim` を「キーバインドモード」と呼ぶ。

混同を避けるため、Vim内の `Normal / Insert` は「Vim操作状態」と呼ぶ。

## 8.5 キーバインド設定要件

### 概要

dayborad はキーボード操作を重視するため、ユーザーが自分の好みに合わせてキーバインドを選択できることを要件とする。

### 要件

- 初期設定では標準キーバインドを利用する。
- 設定画面から Vimキーバインドに切り替えられる。
- キーバインドモードはユーザー設定として永続化する。
- キーバインド切替後、アプリ再起動後も同じ設定を維持する。
- Vimキーバインド利用時も、主要なアプリ内ショートカットは利用できる。
- 入力中の誤操作を避けるため、Vimキーバインドでは Insert 状態と Normal 状態を明確に区別する。

### 設定画面イメージ

```text
キーバインド

○ 標準
  一般的なショートカットで操作する

● Vim
  h/j/k/l、i、Esc などのVim風操作を使う
```

## 8.6 Vimキーバインド仕様

Vimキーバインドでは、各入力領域に対して以下の操作状態を持つ。

| Vim操作状態 | 内容 |
|---|---|
| Normal | 移動、選択、TODO化、モード切替などの操作を行う状態 |
| Insert | テキスト入力を行う状態 |

### 基本挙動

- Vimキーバインドでは、テキスト領域にフォーカスした直後は Normal 状態を基本とする。
- `i` で Insert 状態に入る。
- `Esc` で Insert 状態から Normal 状態へ戻る。
- すでに Normal 状態で `Esc` を押した場合、ノートモードでは仕事整理モードへ戻る。
- `Ctrl / ⌘ + J` による仕事整理モード / ノートモード切替は、Vimキーバインドでも有効とする。

### Vimキーバインド操作一覧（MVP）

MVPでは、Vimキーバインドを完全な編集エンジンとして作らない。仕事整理モード / ノートモードの移動と、入力状態の切り替えに必要な最小範囲に絞る。

| 操作 | キー | 内容 |
|---|---:|---|
| 左へ移動 | `h` | 左の列、またはカーソル左へ移動 |
| 下へ移動 | `j` | 下の項目、またはカーソル下へ移動 |
| 上へ移動 | `k` | 上の項目、またはカーソル上へ移動 |
| 右へ移動 | `l` | 右の列、またはカーソル右へ移動 |
| Insertへ入る | `i` | 現在位置でテキスト入力を開始 |
| Insertを抜ける | `Esc` | Insert 状態から Normal 状態へ戻る |
| TODO完了切替 | `x` | 選択中のTODOの完了 / 未完了を切り替える |
| ノートモード切替 | `Space n` | ①②③ ⇄ ④ を切り替える |
| TODOへ移動 | `Space 1` | ①TODOにフォーカス |
| 障害へ移動 | `Space 2` | ②障害・詰まりにフォーカス |
| 振り返りへ移動 | `Space 3` | ③振り返りにフォーカス |
| 選択行をTODO化 | `Space t` | ノートモードの選択行をTODO化 |
| 選択行を障害化 | `Space b` | ノートモードの選択行を障害化 |

### Vimキーバインド操作一覧（Post-MVP候補）

以下はMVPでは実装しない。MVP利用後、Vim利用者の要望が強い場合に追加する。

| 操作 | キー | 内容 |
|---|---:|---|
| 先頭へ移動 | `gg` | 現在領域の先頭へ移動 |
| 末尾へ移動 | `G` | 現在領域の末尾へ移動 |
| 行末からInsert | `A` | 現在行の末尾で入力を開始 |
| 下に新規行 | `o` | 現在行の下に行を追加して入力開始 |
| 上に新規行 | `O` | 現在行の上に行を追加して入力開始 |
| 現在行を削除 | `dd` | 現在行または選択項目を削除 |
| 元に戻す | `u` | 直前の編集を取り消し |
| やり直し | `Ctrl + r` | 取り消しをやり直し |
| 検索 | `/` | 現在の日付ノート内を検索 |
| 次の検索結果 | `n` | 次の検索結果へ移動 |
| 前の検索結果 | `N` | 前の検索結果へ移動 |
| 選択行を振り返りへ送る | `Space r` | ノートモードの選択行を振り返りへ送る |
| コマンドパレット | `Space k` | コマンドパレットを開く |

### 注意点

- Vimキーバインドは、完全なVim互換を目指さない。
- 目的は、Vimユーザーがストレスなく移動・入力できることである。
- 複雑なマクロ、レジスタ、ビジュアルブロック、ExコマンドはMVP対象外とする。
- 日本語入力中は、VimキーバインドがIME入力を妨げないようにする。
- `Esc` はIME確定、Insert解除、ノートモード復帰の順に扱い、意図しない画面遷移を防ぐ。

## 9. UX要件

## 9.1 ノート1枚感

- PC画面全体を1枚のノートのように見せる。
- 仕事整理モードでは、①②③以外を表示しすぎない。
- ノートモードでは、④を画面いっぱいに表示する。
- ①②③と④を常時同時表示しない。

## 9.2 モード切り替えの感覚

- ショートカットで即座に切り替わる。
- 切り替えは軽快であること。
- 演出を入れる場合は、下からノートが広がるような動きが望ましい。
- 過剰なアニメーションは不要。

## 9.3 入力のしやすさ

- クリックよりキーボード操作を優先する。
- メモ中にTODO化しても、ノートモードから勝手に離脱しない。
- TODO化、障害化した場合は小さな通知を出す。
- ノートモードから戻った際、追加された項目を軽くハイライトする。

## 9.4 キーバインド切替の分かりやすさ

- 標準キーバインドとVimキーバインドの現在状態を設定画面で明確に表示する。
- Vimキーバインド利用中は、現在のVim操作状態が `NORMAL` / `INSERT` として分かるようにする。
- 操作状態表示は控えめにし、入力の邪魔にならない位置に置く。
- Vimキーバインドを知らないユーザーが誤って切り替えても、設定から標準キーバインドに戻せるようにする。

### 表示例

```text
右下表示：VIM NORMAL
右下表示：VIM INSERT
```

## 9.5 情報量の制御

- 仕事整理モードでは、その日に必要な情報だけを表示する。
- 長期タスクやプロジェクト情報は表示しない。
- UIは過度に多機能にしない。

## 10.7 NoteLineMeta

TODO化、障害化などの状態を保持するため、MVPでは行単位のメタ情報を持つ。

```ts
type NoteLineMeta = {
  id: string
  noteEntryId: string
  lineNumberAtConversion: number
  normalizedLineText: string
  lineHash: string
  lineText: string
  convertedToTodoId?: string
  convertedToBlockerId?: string
  convertedToReflection?: boolean
  convertedAt: string
  createdAt: string
  updatedAt: string
}
```

### NoteLineMetaの扱い

- `lineText` は変換時点の原文スナップショットとする。
- `normalizedLineText` は行頭の記号、前後空白、連続空白を正規化した文字列とする。
- `lineHash` は `noteEntryId + normalizedLineText` をもとに生成する。
- `lineNumberAtConversion` は変換時点の行番号であり、ノート本文の編集後に正確な位置を保証しない。
- ノート本文の編集後に同じ行を見つけられない場合でも、TODO / 障害側は `NoteLineMeta.lineText` を発生元として表示できる。
- `convertedToReflection` はPost-MVP用の予約フィールドとし、MVPでは利用しない。

## 12. 非機能要件

## 12.1 パフォーマンス

- アプリ起動後、すぐに今日のノートを表示する。
- モード切り替えは体感で即時に行われること。
- 入力中に保存処理で引っかからないこと。

## 12.2 信頼性

- 入力内容を失わないことを最優先とする。
- 自動保存を基本とする。
- 保存失敗時は明確に通知する。

## 12.3 セキュリティ

- MVPでは単一ユーザー利用を前提とする。
- 外部同期を行わない場合、認証機能は不要。
- 将来的にクラウド同期を行う場合は、認証、暗号化、バックアップ設計を追加する。

## 12.4 可用性

- オフラインでも利用できることが望ましい。
- ローカルファーストの設計が望ましい。
- 将来的に複数プラットフォームで利用する場合も、入力内容を失わないことを優先する。

## 12.5 キーボード操作互換性

- 標準キーバインドとVimキーバインドを切り替えても、入力データに影響を与えない。
- OS標準のショートカット、IME、日本語入力と競合しにくい設計にする。
- MVPではデスクトップアプリ内ショートカットとして実装し、OS全体のグローバルショートカットは利用しない。
- ブラウザ版を将来実装する場合は、ブラウザ既定ショートカットとの衝突を別途再設計する。

## 13. 技術構成

## 13.1 主要技術構成

MVPは、PC向けの入力体験、ショートカット操作、自動保存を重視しつつ、将来的なクロスプラットフォーム対応を見据えて以下の技術構成を主要スタックとする。

```text
TypeScript
React
Hono.js
React Native
PostgreSQL
Tailwind CSS
CodeMirror
```

### 理由

| 技術 | 理由 |
|---|---|
| TypeScript | UI、API、ドメインロジック、データ型を横断して安全に扱える |
| React | PC向けの仕事整理モード、ノートモード、ショートカット中心UIを作りやすい |
| Hono.js | TypeScriptで軽量なAPI層を構築でき、将来のWeb、デスクトップ、モバイルクライアントから共通利用しやすい |
| React Native | Post-MVPでモバイルやタブレット対応を検討する際に、Reactの設計資産を活かしやすい |
| PostgreSQL | 日次ノート、TODO、障害、ノート行メタ情報をリレーショナルに保持し、将来の同期や検索拡張に備えやすい |
| Tailwind CSS | ノート風UIを素早く作れる |
| CodeMirror | ノート本文とVim風キーバインドのMVP範囲を実装しやすい |

## 13.2 クロスプラットフォーム対応方針

MVPではPC向け体験を優先し、Reactによる画面実装とHono.jsによるAPI層を中心に構築する。React Nativeは初期スコープでモバイルアプリを提供するためではなく、Post-MVPでモバイルやタブレットに展開できるよう、設計上の前提として扱う。

クロスプラットフォーム対応に向けて、以下を意識する。

- `DayNote`、TODO、障害・詰まり、ノート行メタ情報などのドメインモデルをUIフレームワークから分離する。
- PC向けUI固有のショートカット処理を、ドメイン操作と分離する。
- Hono.js APIをクライアント共通の境界とし、React版とReact Native版で同じデータ契約を利用できるようにする。
- PostgreSQLを主データストアとし、将来の同期、バックアップ、検索、複数端末利用に備える。
- モバイル最適化、通知、外部同期はMVPでは実装しない。

## 14. UIデザイン方針

### 14.1 見た目

- 紙ノートに近い余白を持たせる。
- 罫線やカラム線は控えめにする。
- 文字入力が主役になるよう、装飾は少なくする。
- 仕事整理モードでは3カラムを明確にする。
- ノートモードでは広いテキストエリアを主役にする。

### 14.2 操作感

- キーボードだけで主要操作が完結すること。
- 保存ボタンを押さなくてよいこと。
- モード切り替えが速いこと。
- ノートからTODO化しても作業の流れを止めないこと。

## 15. MVP受け入れ条件

MVPは、以下の受け入れ条件を満たせば完成とする。

| ID | Given | When | Then |
|---|---|---|---|
| AC-01 | 当日の `DayNote` が存在しない | アプリを起動する | 当日の `DayNote` が自動生成され、仕事整理モードで表示される |
| AC-02 | 仕事整理モードを表示している | 今日のテーマ、TODO、障害・詰まり、振り返りを入力する | 入力内容が日付に紐づいて保存され、再起動後も同じ内容が表示される |
| AC-03 | 仕事整理モードを表示している | `Ctrl / ⌘ + J` を押す | ノートモードへ切り替わり、会議・打ち合わせ・会話メモを画面いっぱいに入力できる |
| AC-04 | ノートモードを表示している | `Esc` または `Ctrl / ⌘ + J` を押す | 仕事整理モードへ戻り、入力途中のノート本文は失われない |
| AC-05 | ノートモードで1行を選択している | `Ctrl / ⌘ + Enter` または Vimの `Space t` を押す | 選択行からTODOが作成され、ノートモードから離脱せず、変換済みマークと小さな通知が表示される |
| AC-06 | TODO化済みのノート行を再度選択している | TODO化操作を行う | 重複確認が表示され、キャンセル時は新しいTODOを作成しない |
| AC-07 | ノートモードで1行を選択している | `Ctrl / ⌘ + Shift + B` または Vimの `Space b` を押す | 選択行から障害・詰まりが作成され、変換済みマークと小さな通知が表示される |
| AC-08 | TODO化または障害化した元行を後から編集・削除している | 作成済みTODOまたは障害を表示する | 変換時点の行本文スナップショットを発生元として確認できる |
| AC-09 | 仕事整理モードでTODOを選択している | 完了操作を行う | TODOのステータスが `done` になり、もう一度操作すると `todo` に戻る |
| AC-10 | 任意の日付のノートを表示している | 前日、翌日、今日へ移動する | 対象日付のノートが表示され、未作成の日付では `DayNote` が自動生成される |
| AC-11 | 当日の未完了TODOが存在する | 翌日へ持ち越す | 翌日のノートにTODOが作成され、元TODOは `carried` になり、持ち越し元の日付が保持される |
| AC-12 | すでに持ち越し済みのTODOが存在する | 同じTODOを再度持ち越そうとする | 重複した持ち越しTODOは作成されない |
| AC-13 | 保存可能な状態で入力している | 入力後、短い遅延が経過する | 保存状態が「保存中...」から「保存済み」に変わる |
| AC-14 | 保存処理が失敗する | 入力後、保存が実行される | 「保存できませんでした」が表示され、ユーザーが入力内容の未保存状態を認識できる |
| AC-15 | 設定画面を表示している | 標準キーバインドとVimキーバインドを切り替える | 選択したキーバインドモードが保存され、再起動後も維持される |
| AC-16 | Vimキーバインドの仕事整理モードでTODO/障害を選択している | `i` / `Enter` / `a` を押す | 当該アイテムの編集モード（input表示）に入りInsert状態となる。`Enter`で確定、`Esc`でキャンセル、いずれもNormal状態へ戻り選択要素へフォーカスが戻る（hjklが即座に使える） |
| AC-17 | Vimキーバインドの Insert 状態で入力している | `Esc` を押す | Normal 状態へ戻り、ノートモードからは離脱しない |
| AC-18 | Vimキーバインドの Normal 状態でノートモードを表示している | `Esc` を押す | 仕事整理モードへ戻る |
| AC-19 | 日本語IMEで変換中である | `Esc` を押す | まずIME変換をキャンセルまたは確定し、意図せず表示モードを切り替えない |
| AC-20 | Vimキーバインドの Normal 状態で項目またはカーソルが選択されている | `h` / `j` / `k` / `l` を押す | 左、下、上、右の基本移動ができる |
| AC-21 | MVPを単一ユーザー環境で利用している | ノートを作成・編集する | PostgreSQLに保存され、外部同期なしの場合は認証なしで利用できる |
| AC-22 | Post-MVPに分類されたショートカットを押す | MVP版で操作する | 未実装でもMVP未達扱いにしない。ただし入力内容を破壊しない |
| AC-23 | テキスト入力欄以外にフォーカスがある状態で `?` を押す、またはヘッダーのヘルプアイコンを押す | キーバインドガイドを開く | 現在の表示モード（仕事整理/ノート）とキーバインドモード（標準/Vim）に応じたショートカット一覧が表示される。`Esc` / 背景クリック / もう一度 `?` で閉じる。テキスト入力欄フォーカス中は `?` を文字入力として扱いガイドは開かない |

## 17. 注意点

### 17.1 やりすぎないこと

dayborad は、Notion、Asana、Backlog のような総合管理ツールを目指さない。

最初に作るべきなのは、以下に絞った軽量な体験である。

```text
① TODOを見る
② 詰まりを見る
③ 振り返る
④ 必要なときだけメモを開く
```

### 17.2 ④を常時表示しないこと

今回の重要な設計思想は、④を普段は隠しておくことにある。

- 普段は①②③を見る。
- 書きたいときだけ④を開く。
- ④は画面いっぱいに表示する。

これにより、PC画面が「ノート1枚」として機能する。

### 17.3 ノートから整理へ送る動線を最重要視すること

dayborad のコア体験は、以下である。

```text
④に書き散らす
↓
必要な行だけ①TODOや②障害へ送る
↓
①②③で今日の仕事を整理する
```

この動線がスムーズであれば、MVPとしての価値は十分に成立する。

## 18. 要約

**dayborad** は、PC画面を「今日の仕事ノート1枚」として使う日次業務管理アプリである。

通常時は、以下を画面いっぱいに表示する。

```text
① TODO
② 障害・詰まり
③ 振り返り
```

会議や会話のメモを書きたいときだけ、ショートカットで以下を画面いっぱいに表示する。

```text
④ 会議・打ち合わせ・会話メモ
```

このアプリの本質は、タスクを大量に管理することではない。

**今日やることを見て、詰まりを把握し、会議や会話のメモを必要に応じて開き、1日の終わりに明日の一手へつなげること**である。
