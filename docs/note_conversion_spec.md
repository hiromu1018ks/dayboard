# dayborad ノート変換仕様

本書は、[要件定義書 7.8 / 7.9](dayborad_requirements.md) と [US-MVP-009 / US-MVP-010](dayborad_user_stories.md) をもとに、ノート行→TODO/障害変換の正規化ルール・行頭記号やラベル除去・重複判定・変換済みマーク追従を固定する設計契約である。

- 未確定だった論点: TODO/障害テキスト生成時の行頭記号や「TODO化:」等のラベル除去範囲（US-MVP-009 / US-MVP-010 Open Questions）
- 関連: [api_contract.md §9 変換エンドポイント](api_contract.md) / [database_schema.md §3.7 NoteLineMeta](database_schema.md) / [ui_interaction_spec.md §6.2](ui_interaction_spec.md) / [edge_cases.md](edge_cases.md)

## 1. 変換の全体フロー

```text
[ノートモードでカーソル行を選択]
        │
        ▼
1. 行テキスト取得（lineText）
        │
        ▼
2. 空行チェック ── 空 ──► 通知「空行は変換できません」、終了
        │ 非空
        ▼
3. 正規化（normalizedLineText）  ← [§3](#3-正規化ルール)
        │
        ▼
4. ラベル・行頭記号除去（title 生成）  ← [§4](#4-todo障害タイトル生成ルール)
        │
        ▼
5. lineHash 生成  ← [§5](#5-linehash-の生成)
        │
        ▼
6. 重複判定（同 noteEntryId + lineHash + 変換先）  ← [§6](#6-重複判定)
        │
   ├─ 重複あり ──► 409 → 確認ダイアログ（[§7](#7-重複確認ダイアログ)）
   │                ├ キャンセル ──► 終了（TODO作成しない）
   │                └ 別作成（force=1）──► 8へ
   │
   └─ 重複なし ──► 8へ
        │
        ▼
8. TodoItem/BlockerItem + NoteLineMeta を1トランザクションで作成
        │
        ▼
9. 変換済みマーク付与・通知（[§8](#8-変換済みマークの表示と追従)）
```

## 2. 行の特定と行番号の扱い

### 2.1 行番号の基準

- **行番号は 1始まり** とする（CodeMirrorの標準行番号表示と一致、可読性優先）
- `lineNumber` はAPIリクエストに含めるが、**変換後の位置保証には使わない**（要件 10.7「ノート本文編集後に正確な位置を保証しない」）

### 2.2 行の取得単位

- 変換対象は **現在カーソルのある1行のみ**（[ui_interaction_spec.md §6.2](ui_interaction_spec.md)）
- 複数行選択の変換はMVPでは扱わない
- `lineText` は当該行全体（行頭・行末の改行は含まない）

## 3. 正規化ルール

`normalizedLineText` は **重複判定のキー** として使う（要件 10.7）。同一内容かどうかを「見た目の差異」によらず判定するため、以下の正規化を順に適用する。

```text
正規化手順（順序厳守）:
1. 前後の空白（半角・全角スペース、タブ、改行）を trim
2. 連続する空白（半角・全角スペース、タブ）を半角スペース1つに圧縮
3. 全角英数字・全角カタカナは正規化しない（ユーザー意図を尊重）
4. 大文字・小文字は正規化しない（ケース敏感）
```

### 3.1 適用例

| 入力行テキスト | normalizedLineText |
|----------------|---------------------|
| `- 宿題：単価表を確認` | `- 宿題：単価表を確認` |
| `  見積作成  ` | `見積作成` |
| `見積　作成` （全角スペース） | `見積 作成` |
| `見積  作成` （連続半角スペース） | `見積 作成` |
| `部長　承認待ち` （全角スペース） | `部長 承認待ち` |

> **重要:** 正規化は「行頭記号除去」を含まない。正規化は空白正規化のみ。行頭記号やラベルの除去は **タイトル生成** で別途行う（[§4](#4-todo障害タイトル生成ルール)）。これは重複判定キーと表示タイトル生成の責務を分け、行頭記号やラベルの有無を同一性判定に反映するためである（後述 [§6.3](#63-同一性判定の範囲) で再説）。

## 4. TODO/障害タイトル生成ルール

US-MVP-009 / US-MVP-010 Open Questions「行頭記号や『TODO化:』等のラベルをどこまで除去するか」を固定する。

### 4.1 除去対象（順序厳守で適用）

```text
1. 前後の空白を trim
2. 行頭のリスト記号を除去:
     - 先頭の "-" / "•" / "・" / "*" とそれに続く空白（0文字以上）を除去
     - 例: "- 宿題：単価表を確認" → "宿題：単価表を確認"
     - 例: "・部長承認待ち" → "部長承認待ち"
3. 行頭の番号リストを除去:
     - 先頭の "\d+[.)]" とそれに続く空白（0文字以上）を除去
     - 例: "1. 見積作成" → "見積作成"、 "2)確認" → "確認"
4. 行頭のTODO/障害ラベルを除去:
     - 以下のラベル（大文字小文字無視、全角半角無視）とそれに続くコロン（: / ：）と空白（0文字以上）を除去
       "TODO", "TODO化", "やること", "障害", "障害化", "詰まり", "Blocker", "メモ"
     - 例: "TODO化：見積作成" → "見積作成"
     - 例: "やること: 確認する" → "確認する"
5. 連続空白を圧縮（[§3](#3-正規化ルール) と同じ）
6. 前後の空白を再度 trim
```

### 4.2 適用例

| 入力行 | 生成されるTODO/障害タイトル |
|--------|------------------------------|
| `- TODO化：見積作成` | `見積作成` |
| `・部長承認待ち` | `部長承認待ち` |
| `宿題：単価表を確認` | `宿題：単価表を確認` ※ 「宿題：」は除去対象外 |
| `* やること: 確認` | `確認` |
| `1. 見積作成` | `見積作成` |
| `TODO：` （ラベルのみ） | 空になる → [§4.4](#44-空になった場合) でエラー |
| `部長承認待ち` | `部長承認待ち` |

### 4.3 除去「しない」もの

意図的に除去対象外とするもの:

| 対象 | 理由 |
|------|------|
| `宿題：`, `決定事項：`, `確認事項：` 等の汎用見出しラベル | ユーザーの文脈依存。機械的に除去すると意味が変わる |
| 絵文字・装飾記号（`※`, `★`, `✔` 等） | 除去ルールを広げすぎると誤除去リスク。MVPでは冒頭1文字がこれらだけでも残す |
| Markdown装飾（`**太字**`, `[リンク](url)` 等） | MVPでは装飾を維持。Markdown解釈は Post-MVP（要件 16） |

### 4.4 空になった場合

ラベル/記号のみの行（例: `TODO化：` のみ）は、除去後に空になる。この場合:

- **変換エラー** として扱い、`VALIDATION_ERROR`（`fields: [{ field: "title", message: "変換後のテキストが空です。" }]`）を返す
- UI側で通知「この行はTODO化できません（本文が空になります）」
- TODO/NoteLineMetaは作成しない

### 4.5 タイトル最大長

- 生成されたタイトルが200文字を超える場合は、**省略記号「…」を含めて200文字以内** に切り詰めて保存する。具体的には先頭199文字 + `…` とする（APIの `VALIDATION_ERROR` を回避しつつ、ユーザーに全文はノートで見せられる）
- 切詰めした旨を通知で伝える（「長いため200文字に切り詰めました」）

## 5. lineHash の生成

[database_schema.md §3.7](database_schema.md) / 要件 10.7 に従い、`lineHash` は **`noteEntryId + normalizedLineText`** を元に生成する。

### 5.1 アルゴリズム

```text
lineHash = fnv1a_64( noteEntryId + "\n" + normalizedLineText ).toString(16) （hex 16文字）
```

- **FNV-1a（64bit）** を使用。純粋 TypeScript 実装で Node・ブラウザ（Electron renderer 含む）双方で同一結果となる
- 当初は SHA-256（Node `crypto`）を使っていたが、Electron renderer（`nodeIntegration:false`）では `node:crypto` にアクセスできずビルドが失敗するため、外部依存のない FNV-1a へ切り替えた。ドメイン層で1関数を共有しているため、サーバー・クライアント間のハッシュ値の整合性は保たれる
- 暗号強度は SHA-256 に劣るが、lineHash の用途（同一 noteEntryId 内の行重複判定用キー）では実用上十分な一意性を持つ（64bit、衝突確率は無視できる）
- `noteEntryId` を含めることで、**異なるノートエントリ間の同じ行テキストを別物** として扱う（要件 10.7「`noteEntryId + normalizedLineText` をもとに生成」）

### 5.2 同一性判定の範囲

重複は **同じ `noteEntryId` 内** でのみ判定する（要件 7.8「同じ `noteEntryId` 内で同一内容の行を再度TODO化しようとした場合」）。異なる日付のノートの同じ行は別物として扱い、重複扱いしない。

## 6. 重複判定

### 6.1 重複の定義

同じ `(noteEntryId, lineHash)` を持ち、かつ対応する変換先が既に存在する `NoteLineMeta` がある場合、重複とする。

- TODO化の場合: `convertedToTodoId IS NOT NULL` の NoteLineMeta
- 障害化の場合: `convertedToBlockerId IS NOT NULL` の NoteLineMeta

### 6.2 重複時のAPI応答

[api_contract.md §9](api_contract.md) に従い、`409 DUPLICATE_CONVERSION` を返す。

```json
{
  "error": {
    "code": "DUPLICATE_CONVERSION",
    "message": "この行はすでにTODO化されています。",
    "details": {
      "existing": {
        "id": "todo_abc",
        "title": "見積作成",
        "sourceNoteLineMetaId": "nlm_prev"
      }
    }
  }
}
```

### 6.3 同一性判定の範囲（再説）

`normalizedLineText` は行頭記号やラベルを除去しないため、以下は **異なる行として扱う**。

| 行A | 行B | 同一? | 理由 |
|-----|-----|-------|------|
| `- 見積作成` | `見積作成` | 異なる | `normalizedLineText` がそれぞれ `- 見積作成` / `見積作成` になり、lineHash が異なる |

**意図:** 行頭記号の有無で重複判定を変えるのは、ユーザーが意図的に別表現で書き分けた可能性を尊重するため。`- TODO化：見積作成` と `見積作成` は別TODOになる。これは直感に近い。

ただし **全角/半角スペースの差、前後空白の差、連続空白** は正規化されるため、以下は同一:

| 行A | 行B | 同一? |
|-----|-----|-------|
| `見積作成  ` | `見積作成` | 同一（前後空白trim） |
| `見積  作成` | `見積 作成` | 同一（連続空白圧縮） |
| `見積　作成` | `見積 作成` | 同一（全角→半角圧縮） |

### 6.4 重複時の2つの道

要件 7.8「重複確認でキャンセルした場合、新しいTODOは作成しない」「重複確認で作成を選んだ場合、別TODOとして作成」を以下で扱う。

1. **キャンセル:** API呼び出しなし（クライアント側でダイアログ閉鎖）。または 409受領後にそのまま終了。TODO/NoteLineMeta作成なし。
2. **別TODO作成:** `POST /api/day-notes/:date/convert/todo?force=1` で重複チェックをバイパス（[api_contract.md §9](api_contract.md)）。2つ目の TodoItem と NoteLineMeta が作られる。両者は別 `id` を持ち、元行は「複数変換済み」として表示。

## 7. 重複確認ダイアログ

### 7.1 表示内容

```text
┌───────────────────────────────────────────────┐
│ この行はすでにTODO化されています                 │
│                                               │
│ 既存のTODO:                                   │
│   □ 見積作成                                   │
│                                               │
│ 別のTODOとして追加しますか？                    │
│                                               │
│              [ キャンセル ] [ 別TODOとして追加 ] │
└───────────────────────────────────────────────┘
```

- 既存TODO候補のタイトルを表示（`details.existing.title`）
- 障害化の場合も同様（「すでに障害化されています」「既存の障害: ...」）

### 7.2 挙動

- ` Esc ` / `キャンセル` → ダイアログ閉じる、何も作成しない
- `Enter` / `別TODOとして追加` → `?force=1` で再リクエスト
- ダイアログ中はノートモードの他操作を無効化（モーダル）

## 8. 変換済みマークの表示と追従

要件 7.8 / 7.9「変換済みの行にはマークを付ける」。ノート本文を編集してもマークが正しい行に残り続ける必要がある。

### 8.1 マークの種類

| 変換先 | マーク（ガター表示、[ui_interaction_spec.md §6.3](ui_interaction_spec.md)） |
|--------|----|
| TODO化済み | `✓T`（仮） |
| 障害化済み | `✓B`（仮） |
| 両方（同一行を両方に変換） | `✓T ✓B`（両方表示） |

### 8.2 マークの追従ルール（行編集後）

ノート本文を編集して行がずれても、`lineHash` が一致する行にマークを付ける。

```text
[ノート本文編集イベント]
        │
        ▼
1. 各行の normalizedLineText を計算
2. 各行の lineHash を計算（noteEntryId + normalizedLineText）
3. NoteLineMeta テーブル（メモリ上のキャッシュ）と照合:
   - lineHash が一致する行 → マーク表示
   - 一致しない行 → マークなし
4. CodeMirror のガター（行デコレーション）を更新
```

### 8.3 元行が特定できなくなった場合

要件 7.8「ノート本文の編集により元行を特定できなくなった場合でも、TODO側には変換時点の行本文を表示できるようにする」。

- TODO/障害側は `NoteLineMeta.lineText`（原文スナップショット）を表示するため、元行が消えてもTODO自体は影響を受けない
- NoteLineMeta レコード自体は **DBに残り続ける**（ノート本文編集で NoteLineMeta は削除されない）
- ただしガター上のマークは、lineHash が一致する行が無くなるため **表示されなくなる**（自然な挙動）
- ユーザーが再度同じ内容の行を書いた場合、その行にマークが復活する（lineHash 再一致）

### 8.4 変換先アイテムが削除された場合

- TODOを削除 → `note_line_metas.converted_to_todo_id` が `ON DELETE SET NULL` で NULL化（[database_schema.md §3.3/§3.7](database_schema.md)）
- この結果、マークも消える（`convertedToTodoId IS NULL` になるため）
- NoteLineMetaレコード自体は残るが、実質的に「変換履歴なし」と同等

## 9. 発生元（スナップショット）の表示

要件 7.8 / 7.9「発生元として、元のノート行への参照を保持」、AC-08「変換時点の行本文スナップショットを発生元として確認できる」。

### 9.1 保持内容

`NoteLineMeta` に以下を保持（要件 10.7）:

| フィールド | 内容 |
|-----------|------|
| `lineText` | 変換時点の原文（行頭記号含む） |
| `lineNumberAtConversion` | 変換時点の行番号（参考値） |
| `normalizedLineText` | 正規化後テキスト |
| `lineHash` | 重複判定用 |
| `convertedAt` | 変換日時 |

### 9.2 TODO/障害からの発生元表示UI

仕事整理モードでTODOを選択・ホバー時、`sourceNoteLineMetaId` があれば **発生元ポップアップ** を表示:

```text
□ 見積作成
   └ 元ノート行（7/8）:
       "- TODO化：見積作成"
```

- `NoteLineMeta.lineText` を表示
- 元ノート行へジャンプする導線はMVPでは任意（ホバー表示のみで十分）

## 10. 変換済み行の再変換（TODO→障害等）

同一行を TODO化した後に 障害化 する、あるいはその逆は **許可する**（要件に禁止記載なし）。

- TODO化済みの行を障害化: 重複判定は **変換先ごと** に行う（[§6.1](#61-重複の定義)）。`convertedToTodoId` がある行でも `convertedToBlockerId` がなければ障害化は重複扱いしない
- 結果として `NoteLineMeta` 1行に両方の参照が入り、マークは `✓T ✓B` と表示（[§8.1](#81-マークの種類)）

## 12. 実装上の指針（domain層）

[architecture.md §4](architecture.md) のドメイン層に、以下のピュア関数を実装する（フレームワーク非依存）。

```ts
// 正規化（重複判定キー生成）
function normalizeLineText(raw: string): string

// ラベル・行頭記号除去 → TODO/障害タイトル生成
function extractTitle(raw: string): string
// 戻り値が空の場合は呼び出し側で VALIDATION_ERROR 扱い

// lineHash 生成
function computeLineHash(noteEntryId: string, normalizedLineText: string): string

// 重複判定（リポジトリ経由で同 noteEntryId + lineHash の NoteLineMeta を取得）
// → API層で 409 を返すか新規作成か分岐
```

これらはピュア関数として [test_strategy.md](test_strategy.md) の単体テストで網羅する。特に `extractTitle` の除去ルールは [edge_cases.md](edge_cases.md) の変換系エッジケースと合わせて境界値テストを充実させる。
