# dayborad 自動保存仕様

本書は、[要件定義書 7.11 自動保存](dayborad_requirements.md) および [US-MVP-011](dayborad_user_stories.md) をもとに、自動保存の遅延値・状態遷移・リトライ・日付移動/モード切替中の未保存データの扱いを実装者が迷わない粒度に固定する設計契約である。

- 未確定だった論点: 「短い遅延」の具体値（US-MVP-011 Open Questions）、保存中/保存済み/失敗の状態遷移、失敗時の retry、日付移動・モード切替中の未保存データ扱い
- 関連: [api_contract.md §12 保存単位](api_contract.md) / [architecture.md §8](architecture.md) / [ui_interaction_spec.md §10 保存状態表示](ui_interaction_spec.md)
- **最優先要件:** 「自動保存失敗による入力喪失報告 0件」（要件 4.3 成功指標、[dayborad_requirements.md](dayborad_requirements.md)）

---

## 1. 設計目標と原則

| 目標 | 出典 | 本書での扱い |
|------|------|--------------|
| 入力内容を失わないことを最優先 | 12.2 | [§6 フェイルセーフ](#6-フェイルセーフローカルバッファ) |
| 入力中に保存処理で引っかからない | 12.1 | [§3 デバウンス](#3-保存遅延デバウンス) で入力を止めない |
| ユーザーが保存操作を意識しない | 7.11 | 保存ボタンを置かない（[§2 保存対象](#2-保存対象と-trigger)） |
| 保存状態を小さく表示 | 7.11 | [ui_interaction_spec.md §10](ui_interaction_spec.md) |
| 失敗時は明確に通知 | 12.2 | [§5 状態遷移](#5-保存状態の有限状態機械) の `error` |

**原則:** 自動保存はクライアント（Renderer）主導でデバウンス実行する。サーバー（Hono）は保存APIを提供するだけで、クライアント側のタイミング制御には関与しない。

## 2. 保存対象と Trigger

### 2.1 保存対象（要件 7.11）

| 対象 | 保存API | 備考 |
|------|---------|------|
| 今日のテーマ | `PATCH /api/day-notes/:date` (`theme`) | [api_contract.md §4](api_contract.md) |
| TODO（追加/編集/完了/削除） | `POST` / `PATCH` / `DELETE` | [§5](api_contract.md) |
| TODO表示順 | `POST /api/day-notes/:date/todos/reorder` | 同上 |
| 障害（追加/編集/解消/削除） | blockers系 | [§6](api_contract.md) |
| 振り返り3セクション | `PATCH /api/day-notes/:date/reflection` | [§7](api_contract.md) |
| ノート本文 | `PATCH /api/day-notes/:date/note-entry` | [§7](api_contract.md) |
| 変換（TODO化/障害化） | `POST /api/day-notes/:date/convert/*` | 即時保存（後述） |
| 持ち越し | `POST /api/day-notes/:date/carry-over` | 即時保存（ユーザー明示操作） |
| キーバインド設定 | `PATCH /api/settings` | 設定モーダル保存時 |

### 2.2 自動保存の2種類

保存Triggerには2種類ある。扱いが異なるので区別する。

| 種類 | 対象 | 挙動 |
|------|------|------|
| **デバウンス保存** | テキスト編集（テーマ、TODO本文、障害text、振り返り、ノート本文） | 入力停止後、[§3](#3-保存遅延デバウンス) の遅延で保存 |
| **即時保存** | 追加/削除/完了切替/並替/変換/持ち越し/設定変更 | 操作直後に保存（デバウンスしない） |

即時保存の理由: これらは「操作の確定」が自明で、かつ画面状態がリスト構造など冪等性に影響するため、遅延せず直ちにサーバーへ反映する。例えばTODO追加をデバウンスすると、その間に並べ替えや完了操作が来て不整合が起きる。

## 3. 保存遅延（デバウンス）

US-MVP-011 Open Questions「具体的な遅延時間は実装時に決定」を固定する。

### 3.1 採用値

- **デバウンス遅延: 800ms**（最後の入力または編集から 800ms間新たな入力がなければ保存発火）

### 3.2 採用根拠

| 候補 | 採否 | 理由 |
|------|------|------|
| 300ms | × | 速いが、日本語変換の確定間隔や連続入力でAPI発火が多くなりすぎる |
| **800ms** | ○ | 一般的な自動保存（Notion、Google Docs等）の実績値。入力リズムに割り込まず、入力停止を検出できる |
| 2000ms | × | 遅すぎる。800msの2.5倍で、モード切替前のフラッシュ体感が悪化 |

### 3.3 実装上の振る舞い

- デバウンスは **編集対象ごと（per field）に独立したタイマー** を持つ（TODO本文とノート本文が同時に保存されても干渉しない）
- 同じ対象に再入力があったら **タイマーをリセット**（要件「入力後、短い遅延」は最後の入力起点）
- タイマー設定時に `saveStatus` を `idle`（未保存あり）へ
- タイマー発火時に保存実行 → `saving` へ

> **TypeScript実装上の指針:** 編集対象（例: `{type:'theme'}` / `{type:'todo', id:'todo_1'}`）をキーにした `Map<Key, TimerHandle>` でタイマーを管理。対象ごとに独立してデバウンス・状態を持つ。

### 3.4 ノート本文の扱い

ノート本文（`PATCH /api/day-notes/:date/note-entry`）は CodeMirror全文を送る（[api_contract.md §7](api_contract.md)）。CodeMirrorの入力イベントをデバウンス対象とする。800msは CodeMirrorの連続入力にも追従できる。

## 4. 即時保存フラッシュのトリガ

一部操作は保留中のデバウンスを待たず、即座に保存を発火する（US-MVP-011 AC-5「モード切り替えや日付移動で入力内容を失わない」）。これを **flush** と呼ぶ。

### 4.1 flush を発火する操作

| 操作 | 出典 | flush内容 |
|------|------|-----------|
| モード切替（`⌘/Ctrl+J`、`Esc`） | US-MVP-008 | 全保留中デバウンス |
| 日付移動（前日/翌日/今日） | US-MVP-002 | 全保留中デバウンス |
| TODO化・障害化（変換） | US-MVP-009/010 | 対象ノート本文の保留 |
| 持ち越し | US-MVP-012 | 全保留 |
| アプリ終了・リロード | 12.2 | 全保留 |
| 設定モーダル保存 | US-MVP-014 | 設定のみ即時（他は保留のままでよい） |

### 4.2 flushの実装

```text
1. 全デバウンスタイマーをクリア
2. 各保留中の編集対象を `dayborad:pending:${date}` の対象別スナップショットへ同期書き込みする
3. 対象別スナップショットの書き込み成功をもって、画面遷移・モード切替を許可する
4. サーバー保存APIは直ちに発火するが、レスポンスや指数バックオフ完了を遷移の完了条件にしない
```

> **UX上の注意:** モード切替・日付移動はローカル永続化を完了条件とし、API/DBの一時停止や 1s→2s→4s のリトライで長時間ブロックしない。サーバー保存に失敗した対象は `error` 表示とバックグラウンドリトライへ回す。localStorageへの同期書き込み自体に失敗した場合のみ、入力喪失リスクがあるため遷移を止めてユーザーへ通知する。

## 5. 保存状態の有限状態機械

[ui_interaction_spec.md §10](ui_interaction_spec.md) と対応する4状態のFSM。

```text
                         編集開始
      ┌─────────────────────────────────────┐
      │                                     ▼
   ┌──┴──┐  タイマー発火/即時   ┌────────┐
   │ idle│ ─────────────────► │ saving │
   │(未保)│                    └───┬────┘
   └─────┘                        │
      ▲                            ├─ 成功 ──► ┌────────┐
      │                            │           │ saved  │
      │ 編集開始（savedから再編集）  │           └────┬───┘
      │                            │                │ 編集開始
      │                            └─ 失敗 ─► ┌─────┴──┐
      │                                        │ error  │
      │                                        └───┬────┘
      │                       リトライ成功           │
      └─────────────────────────────────────────────┘
                                 │ リトライ上限到達
                                 ▼
                          (errorのまま、ユーザー通知)
```

### 5.1 各状態の定義

| saveStatus | 意味 | 表示（要件 7.11） | ユーザーアクション |
|------------|------|-------------------|---------------------|
| `idle` | 保留中の編集あり（タイマー待機中） | 「下書き」（小さく）または非表示 | そのまま入力続行でよい |
| `saving` | サーバーへ保存要求中 | 「保存中...」 | 待機 |
| `saved` | 最新状態が保存済み | 「保存済み」 | — |
| `error` | 保存失敗、リトライ中または上限到達 | 「保存できませんでした」 | リトライ指示 / 手動保存 |

### 5.2 状態遷移表

| 現在状態 | イベント | 次状態 |
|----------|----------|--------|
| `idle`/`saved`/`saving`/`error` | 編集開始 | `idle`（タイマー開始/再開） |
| `idle` | タイマー発火 | `saving` |
| `saving` | レスポンス 2xx | `saved` |
| `saving` | レスポンス 4xx/5xx/ネットワークエラー | `error` → [§7](#7-失敗時の-retry-とユーザー通知) リトライ |
| `saved` | 編集開始 | `idle` |
| `error` | リトライ成功 | `saved` |
| `error` | リトライ上限到達 | `error`（手動復旧へ） |

> **保存状態の粒度:** `saveStatus` はUI上は単一で表示するが、内部では [§3.3](#33-実装上の振る舞い) の対象ごとの状態を持つ。UI表示は「いずれかの対象が `error` なら `error`、いずれかが `idle`/`saving` ならその優先、全部 `saved` なら `saved`」のように集約する。

## 6. フェイルセーフ（ローカルバッファ）

要件「入力内容を失わないことを最優先」（12.2）を担保するため、サーバー保存に依存しないフェイルセーフ層を置く。

### 6.1 編集内容のローカルバッファ

Rendererは **全編集内容をメモリのローカルバッファ（UI状態そのもの）に保持** し、保存APIはそのバッファからサーバーへ同期する役割のみ持つ。

- 編集 → 即座にローカルバッファ更新 → （デバウンス後に）サーバーへ同期
- この結果、**保存APIが失敗しても入力内容はローカルバッファに残り続ける**。ユーザーが画面から入力を失うことはない。

### 6.2 永続化されたローカルバッファ（必須）

アプリクラッシュや `beforeunload` 中の非同期保存未完了でも入力を復元できるよう、以下の **localStorage フェイルセーフ** をMVP必須要件として実装する（要件 4.3「入力喪失報告0件」の保険）。

- 編集ごと、またはUI状態のローカルバッファ更新直後に、保存対象のスナップショットを `localStorage` のキー `dayborad:pending:${date}` に書き込む。デバウンスはサーバー同期の遅延だけを制御し、永続バッファ書き込みは待たない
- キーは日付単位だが、中身は **保存対象ごと** に `dirty` 情報とpayloadを分ける。テーマ保存だけが成功しても、同じ日付のノート本文やTODOの未同期分を消してはならない
- flushまたはデバウンス発火時には、同じキーのうち該当対象のスナップショットを最新内容で上書きしてからサーバーへ送信する
- サーバー保存成功時は該当対象だけを `targets` から削除する。当該日付の全対象が保存済みになり `targets` が空になった時だけ `dayborad:pending:${date}` キーを削除する
- アプリ起動時に `localStorage` の `dayborad:pending:*` を走査し、未同期データがあれば:
  1. サーバーへ再送（リカバリ）
  2. 成功した対象だけ `targets` から削除
  3. 失敗時は [§7](#7-失敗時の-retry-とユーザー通知) のリトライ状態へ
  4. `targets` が空になった日付キーだけ削除

これにより、アプリクラッシュ・リロード後も未保存データは復元される。

**スナップショット形状（概念）:**

```ts
type PendingSnapshot = {
  version: 1
  date: string
  updatedAt: string
  targets: Record<string, {
    target:
      | { type: 'dayNote'; field: 'theme' | 'lastOpenedMode' }
      | { type: 'todo'; id: string }
      | { type: 'todoOrder' }
      | { type: 'blocker'; id: string }
      | { type: 'reflection' }
      | { type: 'noteEntry' }
    payload: unknown
    dirty: true
    updatedAt: string
    lastError?: string
  }>
}
```

`targets` のキーは `noteEntry`、`dayNote:theme`、`todo:todo_1` のように対象を一意に表す文字列とする。POST系の冪等化が必要な対象は `payload` 内にリクエストIDを含める（[§8.2](#82-冪等性)）。

> **容量上限:** localStorageは数MBが上限。ノート本文全文が大きくなりうるため、キーは `dayborad:pending:${date}` のように日付単位で分け、超過リスクを減らす。本MVPの単一ユーザー・日次利用では実用上問題ない。

## 7. 失敗時の Retry とユーザー通知

### 7.1 リトライポリシー

保存API呼び出しが失敗した場合（4xx/5xx/ネットワークエラー）、以下のリトライを行う。

| 項目 | 値 | 根拠 |
|------|-----|------|
| 最大リトライ回数 | **3回** | 過剰なリトライでサーバーを叩きすぎない |
| リトライ間隔 | **指数バックオフ: 1s → 2s → 4s** | ネットワーク・サーバー復旧を待つ |
| リトライ対象エラー | ネットワークエラー、500、503、504（一時的） | 4xx（バリデーション等）はリトライしない（冪等でないため） |

```text
失敗 ──► 1s待 ──► retry1 ──► 2s待 ──► retry2 ──► 4s待 ──► retry3 ──► 上限到達
                                                                         │
                                                                         ▼
                                                              saveStatus = error (最終)
                                                              ユーザー通知（[§7.2](#72-ユーザー通知)）
```

### 7.2 ユーザー通知

リトライ上限到達で `saveStatus = error` に遷移したら:

- 画面右下の保存状態表示を **「保存できませんでした」**（赤）
- その下に **「再試行」** ボタンを表示（手動リトライ）
- トーストで **「保存に失敗しました。入力内容は保持されています。再試行してください。」** を1度だけ表示
- ローカルバッファ（[§6](#6-フェイルセーフローカルバッファ)）にデータは残っていることをユーザーが認識できる文言

「再試行」ボタンは、保留中の編集すべてを対象に再保存する。以降、ユーザーが編集を続けると新規デバウンスが走り、そちらは通常の成功パスに戻る。

### 7.3 部分失敗の扱い

複数対象（テーマ + TODO本文 + ノート本文）が同時に保存されて一部だけ失敗した場合:

- 失敗した対象のみ `error` 状態、成功した対象は `saved` へ
- UI表示は「いずれか `error` があれば `error` を優先表示」（[§5.2](#52-状態遷移表)）
- 「再試行」は `error` 状態の対象のみ再送

## 8. 競合制御（楽観的更新と冪等性）

### 8.1 楽観的更新

- 編集は **即座にローカルバッファとUIへ反映**（レスポンスを待たない）。これは要件 12.1「入力中に保存処理で引っかからない」のため
- 保存APIのレスポンスが返った時点で、ローカル状態と突き合わせ:
  - サーバー側の `updatedAt` を受け取り、最新版として取り込む
  - サーバー側の正規化（例: テーマ空文字→`null`）をローカルへ反映

### 8.2 冪等性

- `PATCH` リクエストは部分更新であり、ネットワーク再送が起きても同じ結果になる（冪等）
- ただし POST（TODO追加など）は冪等でない。リトライで**二重作成**を避けるため、クライアント側で「リクエストID（UUID）」を生成し、サーバー側で短時間（例: 60秒）の重複排除を行う設計余地を残す（MVPでは実装しなくてもよいが、TODO二重追加が起きた場合は[edge_cases.md](edge_cases.md) の対処へ）

### 8.3 更新競合

MVPは単一ユーザー・単一プロセス（[architecture.md C7](architecture.md)）のため、他クライアントとの競合は起きない。よって `updatedAt` ベースの楽観ロックは導入しない。将来の複数端末同期では別途設計する。

## 9. モード切替・日付移動中の未保存データ（US-MVP-011 AC-5）

要件「モード切り替えや日付移動で入力内容を失わない」を保証する手順を明文化する。

### 9.1 モード切替時

```text
1. ユーザーが ⌘/Ctrl+J を押下
2. Renderer: 全保留デバウンスをflushし、最新状態を対象別localStorageスナップショットへ同期書き込み
3. localStorage書き込み成功後、viewModeを即時切替、ノート/仕事整理画面を描画
4. サーバー保存はバックグラウンドで継続し、失敗時は [§7](#7-失敗時の-retry-とユーザー通知) の状態へ
5. localStorage書き込み自体が失敗した場合のみ、切替を保留してユーザーへ通知
```

### 9.2 日付移動時

```text
1. ユーザーが Alt+→ 等を押下
2. Renderer: 当日の全保留デバウンスをflushし、最新状態を対象別localStorageスナップショットへ同期書き込み
3. localStorage書き込み成功後、GET /api/day-notes/:targetDate/full で次の日付データを取得
4. currentDate と全UI状態を差し替え（[ui_interaction_spec.md §7](ui_interaction_spec.md)）
5. 移動元の未保存データは `dayborad:pending:${date}` に残り、サーバー保存が未完了でも起動時/復帰時に復元可能
6. サーバー保存はバックグラウンドで継続し、失敗時は [§7](#7-失敗時の-retry-とユーザー通知) の状態へ
```

### 9.3 localStorage書き込み失敗時のフェイルセーフ

遷移前のlocalStorageスナップショット書き込みに失敗した場合、**日付移動・モード切替を中断** し、ユーザーに確認する。サーバー保存のリトライ失敗だけでは遷移を止めない。

```text
「未保存データをローカルに保護できませんでした。このまま移動すると入力内容が失われる可能性があります。
 [移動する] [キャンセル]」
```

- 「キャンセル」: 現在画面に留まり、メモリ上のローカルバッファのデータは維持される
- 「移動する」: ユーザー明示の上で遷移する。localStorageには保護できていないため、クラッシュ時の復元保証はない

## 10. アプリ終了時の保存保証

### 10.1 正常終了

Electron mainプロセスが `before-quit` イベントを受けたら:
1. Rendererへ `flush-all` を要求
2. Rendererが全保留対象をlocalStorageへ同期書き込みするまで待機
3. 終了

ただし `before-quit` の待ち時間には上限がある（Electronの仕様）。確実な保存のため、[§6.2](#62-永続化されたローカルバッファ必須) の localStorageバッファが**真の保険**になる。

### 10.2 異常終了・クラッシュ

- localStorageバッファ（[§6.2](#62-永続化されたローカルバッファ必須)）に未保存分が残る
- 次回起動時にリカバリ（[§6.2](#62-永続化されたローカルバッファ必須) の起動時走査）

### 10.3 `beforeunload`

Rendererの `beforeunload` で `flush-all` を発火し、まずlocalStorageへ対象別スナップショットを書き込む。その後、可能な限りサーバー保存を試みる。ただし非同期保存が完了する前にアンロードされるリスクは localStorageバッファで補完する。

---

## 11. テスト観点（test_strategy.md への引き渡し）

[test_strategy.md](test_strategy.md) で具体化するが、本仕様の主要テスト対象を列挙する。

| 観点 | 単体/結合/E2E | 内容 |
|------|----------------|------|
| デバウンス800ms | 単体 | 入力後800msで保存発火、その間の再入力でタイマーリセット |
| 即時保存（追加/削除/完了） | 結合 | 操作直後にAPI呼び出し |
| flush on モード切替 | E2E | 保留中編集が切替前にlocalStorageへ対象別保存され、サーバー失敗中でも切替が長時間ブロックされない |
| flush on 日付移動 | E2E | 同上 |
| リトライ（指数バックオフ） | 単体 | 3回まで、1s→2s→4s |
| リトライ上限で error | 結合 | `saveStatus=error`、再試行ボタン表示 |
| localStorageリカバリ | 単体 | 異常終了後、起動時に対象別の未保存分を復元 |
| IME入力中の保存Trigger | E2E | 日本語変換中に保存が発火しない（[ui_interaction_spec.md §9](ui_interaction_spec.md)） |
| 二重POST回避 | 結合 | TODO追加のリトライで二重作成されない（実装依存） |
